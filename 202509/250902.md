### [WORK] 주문번호 dup 에러 디버깅
주문번호는 unique 값인데 동일한 주문번호로 주문이 들어와 `DataIntegrityViolationException`이 발생하는 현상을 추적했다.

- 주문번호가 중복인 데이터가 들어오면 예외가 발생하는 것 자체는 자연스러운 현상. 다만 왜 이런 일이 발생하는지 알아야 후속 조치(클라이언트 중복 호출 방지 / API 단에서 조회 후 무시 / API 단에서 조회 후 예외 처리)를 정할 수 있어 원인을 추적함.
- 로그에 남은 주문번호로 주문 내역·주문자 데이터를 추출했으나 데이터상 교집합이 없어, 특정 상황이 아니라 주문 자체 흐름에서 발생한 문제로 판단.
- 호출 흐름을 살펴보니 주문 페이지에서 [결제하기]를 누르고 주문 완료 페이지로 넘어갈 때 발생한 것으로 추정. 네트워크 지연으로 새로고침을 여러 번 하여 요청이 동시에 이루어졌을 가능성을 의심.
  - 재현을 위해 브라우저 속도를 저하시킨 뒤 결제하기를 누르고 무한 새로고침을 시도했으나, 클라이언트(PHP)가 이미 세션으로 완료 여부를 관리하고 있어 얼럿만 뜨고 API가 동시 호출되지는 않음을 확인.
- 중복 호출 판단 로직을 다시 살펴보니, PHP 세션에 플래그를 저장하고(`PHPSESSID`) 해당 플래그가 있으면 재호출하지 않는 방식이었음. 이를 단서 삼아 쿠키의 `PHPSESSID`를 제거하고 새로고침하니 동일 에러가 재현됨.
- 재현 성공 후 에러 발생 시각과 주문번호 실제 생성 시각을 비교한 결과, 모두 주문번호 생성 이후 시점에 에러가 발생. 주문 완료 후 해당 페이지를 떠나지 않은 채 시간이 지난 뒤 재접속·새로고침하여 발생한 것으로 결론 지음.
