## 🔍️ 오늘 무엇을 했나요? Review
### [WORK] 패키지 대여 정책 지원 

아침에 출근하자마자 예약된 메시지가 도착했다. 당장 월요일에 진행되는 이벤트가 있는데 기능이 있는 줄 알고 열었으나 뒤늦게 해당 기능이 없다는 걸 알았다는 내용이었다. 
패키지를 대여할 수 있는 기능이었는데, 해당 상품을 기획할 때 패키지의 경우 소장만 가능하다고 정책을 세웠고 그렇게 구현이 되어있었다. 
월요일에 동작해야한다면 최소 오전에 기능 구현을 마무리하고 오후에는 QA + 배포까지 해야하는 상황이었다. 불가피하게 주말 출근을 한다던가.. 사실 이벤트를 잠시 미뤄도 되지만 이미 홍보된 것들이 있기 때문에 이것은 제일 나중의 선택지이다. 

우선, 이미 알고 있지만 예상 못한 오류를 범하지 않기 위해 기존 로직이 어떻게 되어있는지 살펴봤다. 그러고 하드코딩이 없으며 기존 로직에 예외 상황을 만들지 않고 (원래 동작하던 기능이 정상 동작) 최소의 변경으로 처리할 수 있는 방법은 떠올렸다. 
콘텐츠를 사용자에게 입과시키는 과정과 장바구니/주문/구매 영역은 분리되어있어 외부에서 걱정하는 것보다는 훨씬 더 간단한 방법으로 처리할 수 있겠다는 생각이 들었다. 

우선, 해당 가설이 맞는지 확인하기 위해 mock 데이터를 만들어서 실행했고 정확히 가설대로 동작했다. 이후 빠르게 해당 작업을 완료했다. 
최소한의 작업, 사용자가 해당 상품을 구매했을 때 대여로 콘텐츠가 입과되는 요구 조건만 충족하도록 처리했다. 이것을 처리한다고 패키지 대여 기능이 완벽하게 동작하는 것은 아니다. 

운영자가 패키지 상품을 등록할 때 대여로 등록할 수 있어야하고 가격 검증 로직 등등 수정해야 할 곳은 더 많지만 그건 사용자가 주문/구매 -> 대여로 입과를 하는데는 중요하지 않다. 

빠르게 개발 서버에서 기능 QA를 혼자 진행하고 이상 없는 것을 확인하여 팀원들에게 기존 기능이 원래 시나리오대로 정상 동작하는지 (여러 상품군, 여러 결제수단, 여러 가격(유료/무료)) 케이스를 나눠서 QA를 진행했다. 그 과정에서 미처 고려하지 못한 케이스도 발견하고 기존에 있던 버그도 발견하게되어 모두 수정해 배포했다. 

빠르게 처리할 수 있었던 것은 기존 로직에서 잘 분리를 해놨던 것. 어떤 문제가 해결되야하는지 가장 중요한 부분이 무엇인지 문제를 최소화하려고 했던 부분이 가장 크게 작용한거 같다.
